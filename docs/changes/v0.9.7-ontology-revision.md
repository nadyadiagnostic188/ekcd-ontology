# EKCD v0.9.7 Ontology Revision Dossier

## 1. Summary

This document records the rationale, scope, and implementation basis of the first public release of **EKCD: EKC Derived Ontology**.

EKCD is a personal derivative ontology line developed by **Dong Shin SEO** on the basis of the previously released EKC ontology.  
It was established in order to separate:

- the **institutional EKC baseline**, originally released and maintained with institutional creator metadata, and
- the **ongoing personal extension and refinement work** carried out after that release line.

This version therefore marks the formal establishment of **EKCD** as a separate ontology namespace, repository, and release track.

---

## 2. Background

The earlier EKC ontology release line originated from an ontology schema that had first existed in a wiki-based form, centered mainly on classes and object properties.  
That schema was later comprehensively reviewed, formalized, and maintained in **OWL/Turtle** through Protégé-based work, followed by public release and dissemination processes including:

- GitHub releases
- ontology upgrades
- LOV registration and maintenance
- Zenodo DOI publication

Those earlier public release processes were associated with institutional creator metadata reflecting the research institute context.

Subsequent ontology expansion work, however, required a separate release line in which the creator should be recorded personally rather than institutionally.  
EKCD was therefore established to provide a distinct ontology track for those later personal extensions.

## 5. Main Changes Introduced in v0.9.7

### 5.1 Establishment of a Separate Derivative Ontology Line
The most important change in this version is not merely the addition of terms, but the establishment of a **separate derivative ontology identity**:

- separate repository
- separate namespace
- separate release track
- separate creator metadata

### 5.2 Added Datatype Properties
The following datatype properties are introduced under the EKCD namespace for practical dataset construction:

- `ekcd:occurrenceDate`
- `ekcd:startDate`
- `ekcd:endDate`
- `ekcd:originalDate`
- `ekcd:accessURL`

These properties were introduced to improve the explicit modeling of temporal information and web resource access in spreadsheet-driven and Cellfie-based RDF production workflows. `ekcd:occurrenceDate` supports the recording of dates for single-point events, whereas `ekcd:startDate` and `ekcd:endDate` provide a clearer structure for representing temporal spans. `ekcd:originalDate` preserves original or source-oriented date expressions that may not conform to normalized formats, and `ekcd:accessURL` enables more explicit representation of access-related web information for digital or online resources.

### 5.3 Mapping-Related Design and Operational Guidance
As part of the restoration of EKC to its original design, EKC itself uses `owl:sameAs` as its sole property for identity or equivalence-oriented linking.

EKCD, however, adopts additional standard SKOS mapping properties in order to support a more differentiated treatment of correspondence relations in extended ontology and dataset work:

- `owl:sameAs`
- `skos:exactMatch`
- `skos:closeMatch`
- `skos:mappingRelation`

These are adopted as standard vocabulary terms and are not reminted under the EKCD namespace.  
Instead, EKCD provides operational guidance for their stricter and more explicit interpretation in practice.

Within this framework, `owl:sameAs` is reserved for strict identity, `skos:exactMatch` for near-equivalent correspondence without full identity assertion, and `skos:closeMatch` for looser but still meaningful similarity. `skos:mappingRelation` is treated as a broader mapping predicate for cases where a more specific SKOS mapping relation is not yet determined or not required.

This arrangement preserves the original simplicity of EKC while allowing EKCD to introduce a more rigorous distinction between identity, exact correspondence, and close correspondence.

> **Editorial note.**  
> The distinction among `owl:sameAs`, `skos:exactMatch`, and `skos:closeMatch` is operationally significant for link curation. In particular, EKCD avoids using `owl:sameAs` as a general-purpose similarity predicate. Instead, it reserves `owl:sameAs` for cases of strict identity and employs SKOS mapping properties where correspondence is strong but not ontologically identical or where similarity remains interpretive rather than fully identical.

## 6. Items Deliberately Separated from the Institutional EKC Line

Some modeling elements and mapping-related operational additions had not appeared in the original institutional wiki-based ontology schema and were introduced later in the course of personal maintenance and ontology refinement work.

For governance and release clarity, such elements are treated in this release line as part of **EKCD**, not as part of the institutional EKC baseline.

This separation is especially important for:

- creator attribution,
- namespace clarity,
- future maintenance responsibility,
- distinction between institutional and personal release authority.

## 7. Use in Dataset Construction

This release is intended to serve as the active TBox for ongoing dataset production, including spreadsheet-driven workflows using **Protégé + Cellfie**.

In particular, it is expected to support:

- ABox dataset construction
- external links dataset construction
- explicit recording of single-point dates, temporal ranges, and original or source-oriented date expressions
- more explicit representation of access-related web information for digital and online resources, including through `ekcd:accessURL`
- more explicit operational control over external identifier mapping and link curation

Accordingly, this release is designed to support not only routine RDF generation from spreadsheet inputs, but also a more explicit and operationally robust treatment of temporal information, access URLs, and mapping relations in EKCD-based dataset production.

A key immediate use case is the construction of the **국조속오례의**-related dataset.

The significance of this release lies not only in ontology revision itself but also in its operational consequences for dataset construction. In practice, the added temporal properties, the introduction of `ekcd:accessURL`, and the stricter differentiation of mapping predicates require a clearer modeling discipline in spreadsheet-driven and Cellfie-based RDF production. The following recommendations therefore summarize the intended modeling practice for EKCD-based dataset work.

---

## 8. Recommended Modeling Practice

### 8.1 Date Modeling
- use `ekcd:occurrenceDate` for single-point events
- use `ekcd:startDate` and `ekcd:endDate` for period events
- use `ekcd:originalDate` for source-form date expressions

### 8.2 Web Access Information
- use `ekcd:accessURL` to represent access-related web information for digital or online resources
- record `ekcd:accessURL` where explicit documentation of an access URL is operationally useful in spreadsheet-driven and Cellfie-based workflows
- do not confuse `ekcd:accessURL` with identity, equivalence, or mapping predicates

### 8.3 External Mapping
- use `owl:sameAs` only for strict identity
- use `skos:exactMatch` for strong conceptual correspondence
- use `skos:closeMatch` for looser conceptual correspondence
- avoid treating `skos:mappingRelation` as the default data-entry relation

### 8.4 Dataset Separation
Maintain separate outputs for:
- **ABox TTL**
- **links TTL**

This improves maintainability, clarity, and downstream review.

## 9. Template and Workflow Implications

For spreadsheet templates and Cellfie-based workflows, this release implies at least the following updates:

- add `ekcd:` to the prefix environment
- add EKCD temporal properties and `ekcd:accessURL` to relevant property picklists
- reflect the usage rules for date properties and `ekcd:accessURL` in precheck sheets
- distinguish temporal properties, access-related web information, and external mapping properties more explicitly at the template level
- keep external mapping properties explicitly distinguished
- continue using a maintainer-side review sheet such as `Merge_Check`

More specifically, spreadsheet templates should be revised so that `ekcd:occurrenceDate`, `ekcd:startDate`, `ekcd:endDate`, `ekcd:originalDate`, and `ekcd:accessURL` are available for controlled selection and review. This requires corresponding adjustments not only to property picklists but also to precheck logic and maintainer-side review procedures, so that single-point dates, temporal ranges, source-form date expressions, access-related web information, and mapping assertions are handled in a differentiated and operationally consistent way.

The broader workflow should remain compatible with the previously established structure centered on `Entities`, `Triples`, `ExternalIDs`, and maintainer review logic.

### 9.1 Prefixes
The spreadsheet prefix environment should be updated to include the `ekcd:` prefix and its associated namespace URI. This change is necessary so that newly introduced EKCD properties can be referenced consistently across template sheets, Cellfie transformations, and downstream RDF output.

### 9.2 Picklists
Relevant property picklists should be extended to include the new EKCD properties for temporal description and web access information, namely `ekcd:occurrenceDate`, `ekcd:startDate`, `ekcd:endDate`, `ekcd:originalDate`, and `ekcd:accessURL`. At the same time, mapping-related picklists or guidance areas should preserve explicit distinction among `owl:sameAs`, `skos:exactMatch`, `skos:closeMatch`, and `skos:mappingRelation`, rather than treating them as interchangeable data-entry options.

### 9.3 Triples
The `Triples` sheet should be revised so that the newly introduced EKCD properties can be selected and used in a controlled manner during data entry. In operational terms, this means supporting the differentiated use of:
- `ekcd:occurrenceDate` for single-point events,
- `ekcd:startDate` and `ekcd:endDate` for temporal spans,
- `ekcd:originalDate` for source-form date expressions, and
- `ekcd:accessURL` for access-related web information associated with digital or online resources.

This also implies that web access information should not be conflated with identity or mapping assertions, and that access URLs should be recorded as descriptive data rather than as RDF equivalence links.

### 9.4 Protege_Precheck
Precheck logic should be updated to reflect the intended usage rules of the new EKCD properties. In particular, the review sheet should help distinguish:
- single-event date recording from period-based date recording,
- normalized temporal values from source-oriented date expressions,
- access-related web information from external mapping assertions, and
- strict identity links from broader or weaker correspondence relations.

Accordingly, precheck criteria should not only test whether a property is syntactically present, but also whether it is being used in a semantically appropriate way within the intended workflow.

### 9.5 Merge_Check
Maintainer-side review sheets such as `Merge_Check` should continue to serve as the operational checkpoint for consistency control before RDF export or integration. In the revised workflow, this review should explicitly cover:
- whether EKCD temporal properties are used according to their intended scope,
- whether `ekcd:accessURL` is used for access-related web information rather than for identity or mapping purposes,
- whether external mapping properties remain clearly differentiated, and
- whether links-oriented assertions are kept separate from the main ABox description where appropriate.

### 9.6 ExternalIDs and Dataset Separation
The broader template structure remains compatible with the established separation between descriptive ABox construction and external linking work. In this framework, the `ExternalIDs` sheet and related review logic should continue to support links-oriented data management, while final outputs remain separated into:
- **ABox TTL**
- **links TTL**

This separation continues to improve maintainability, clarity, and downstream review, while allowing EKCD-based workflows to handle dates, access URLs, and mapping relations with greater explicitness.
